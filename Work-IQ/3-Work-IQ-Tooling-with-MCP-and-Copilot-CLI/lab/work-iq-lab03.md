# Lab 03: Tooling with MCP & Copilot CLI


## Advanced Scenarios with Work IQ in GitHub Copilot CLI

You can run Work IQ inside GitHub Copilot CLI via MCP protocol.

Run the following command, to start GitHub Copilot CLI:

```
copilot
```

If prompted, trust GitHub Copilot to access the files in your current folder.

Now, run the following prompt:

```
What are my meetings for next week? List all of them sorted by date and help me prepare for them.
```

GitHub Copilot will analyze your prompt and determine that the **ask_work_iq** MCP tool is needed to provide you with an answer. As such, it will ask your consent to use the tool. You can say **1. Yes** to consent it just once, or **2. Yes, and don't ask again for tool "ask_work_iq" from "workiq" in this repo** if you want to consent it from now on. Of course, you can also choose **3. No** and avoid using Work IQ.

If you consented, GitHub Copilot will start processing your request and will talk with the Work IQ MCP server to extract to requested information.

The response will be a list of events in your calendar for next week, including some useful information to prepare for them.

Now let's try with something more complex. Provide the following prompt to GitHub Copilot:

```
Generate a MD document with the information you just retrieved
```

GitHub Copilot will ask your consent to save the Markdown document in the local folder where you are running it.

## Complex Productivity Scenario with Work IQ and GitHub Copilot CLI

### Setting the initial context

Now, imagine that you have received an email from one of your colleagues. The email informs you about a customer who is requesting a prototype of a solution, based on a set of functional and technical requirements that your colleague is sharing with you.

For the sake of simplicity, send yourself an e-mail with subject **"New solution for project management in Zava Insurance"** and use the following body for the message:

```
Hi,
We need to create a new software solution for the IT department of Zava Insurance.
The software goal is to help people manage projects.

Here are the functional requirements:

- Handling of projects (CRUDQ)
- Every project is made of the following information:
    - Project Code (autogenerated)
    - Customer (string)
    - Description (multiline text)
    - Start date (date)
    - Estimated end date (date)
    - Actual end date (date)
    - Estimated effort (work days)
    - Status (enum: Pending approval, Under construction, Released, Canceled)
    - Notes (multiline text)
- There should be a main page with all the project grouped by status and sorted by Estimated end date (those ending sooner should come first)
- There should also be evidence of project that are late on schedule (over the estimated end date or with an actual end date later than the estimated end date)
- From the main page it should be possible to add new projects or to update already existing ones
- When editing an existing project, it will also be possible to "Cancel" a project, but not to delete it. A project, once it is created, cannot be deleted
- Feedback form to collect user's feedback, to be stored in the persistence storage of the solution

Here are the technical requirements:
- Persistence storage on Azure Storage Tables. Use local azurite for development purposes in local dev environment
- Single page web application with TypeScript/Node.js/React
- Nice looking modern UI
- Zava logo, configurable in the application settings, should show up in the header of every page, in the upper left corner

Please, create a prototype to discuss with the customer in the next weekly meeting!
```

Wait few minutes for the e-mail to get delivered and processed by the Semantic Index of Microsoft 365 Copilot. To check if the e-mail message was processed and indexed, you can try executing the following prompt:

```
Can you find in my work related inbox an e-mail that I sent to myself regarding a prototype for Zava Insurance?
```

Once you will be able to retrieve the content of the e-mail, you are good to proceed with the lab.

### Leveraging vibe coding and Work IQ in GitHub Copilot CLI

Now, you can ask GitHub Copilot to build the prototype for you, reading the requirements in the e-mail and scaffolding for you the solution. You can use a prompt like the following one:

```
Help me create a PoC of the solution described in the e-mail that I've got in my work inbox and that I've just sent to myself. The e-mail is about a new project for Zava Insurance. Scaffold the solution and develop the prototype for me! Also create some mock data to allow me testing that the PoC is fully functional.
```

This prompt will require GitHub Copilot to consume the **ask_work_iq** MCP tool to search and retrieve the body of the e-mail. Then it will work with its powerful vibe coding capabilities to create the prototype for you. It should take about 10 minutes or so to go through the whole process. At the end, you will have a fully functional prototype of the solution described in the hypotethical e-mail.

Now image how powerful it is to being able to connect the intelligence of GitHub Copilot with the intelligence of Microsoft 365 Copilot. For example, in a real life scenario, you could have a Microsoft Teams meeting with a customer to discuss an ongoing software project. You could record the meeting and  generate the meeting summary via Microsoft 365 Copilot. Then, you could ask GitHub Copilot to process all the customer feedback, updating the codebase of the project accordingly to the customer's requests. Isn't it super powerful?
